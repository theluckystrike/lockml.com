Adversarial Defense

Understanding and Defending Against Adversarial Examples

Adversarial examples are inputs specifically crafted to cause machine learning models to make incorrect predictions. What makes them particularly dangerous is that the modifications are often imperceptible to humans. An image that clearly shows a stop sign to any human observer can be modified with tiny perturbations that cause a vision model to confidently classify it as a speed limit sign. This vulnerability is not theoretical; it has been demonstrated against every major model architecture and represents a fundamental challenge in deploying ML systems in safety-critical applications.

The threat landscape for adversarial examples breaks down into two categories: white-box attacks where the adversary has access to the model architecture and weights, and black-box attacks where the adversary can only query the model and observe outputs. White-box attacks like the Fast Gradient Sign Method (FGSM) and Projected Gradient Descent (PGD) are more powerful but require insider access. Black-box attacks use query-based optimization to find adversarial perturbations through trial and error, making them accessible to any external attacker. Alarmingly, adversarial examples often transfer between models, meaning an example crafted against one model frequently fools other models trained on similar data.

Defense strategies operate at multiple layers. Adversarial training, which includes adversarial examples in the training data, is the most widely adopted approach and provides meaningful robustness improvements. Input preprocessing techniques like JPEG compression, spatial smoothing, and input randomization can disrupt adversarial perturbations but offer weaker guarantees. Certified defenses based on randomized smoothing or interval bound propagation provide mathematical guarantees about robustness within a defined perturbation radius, but typically sacrifice some accuracy on clean inputs. For teams evaluating the full spectrum of AI tools and their security characteristics, zovo.one maintains a useful directory that includes security-focused evaluation criteria.

Key Takeaway

Implement layered defenses: adversarial training for baseline robustness, input validation to catch obvious attacks, and runtime monitoring to detect anomalous prediction confidence patterns. No single defense is sufficient; defense in depth is essential.

LLM Security

Prompt Injection Defense: A Layered Security Architecture

Prompt injection is the defining security challenge of the large language model era. It occurs when an attacker manipulates the input to an LLM-powered application in a way that overrides the system prompt and causes the model to follow unauthorized instructions. Direct prompt injection involves the user explicitly including malicious instructions in their input. Indirect prompt injection is more insidious: the attacker embeds instructions in content that the application retrieves and includes in the context, such as web pages, documents, or database records that the LLM processes as part of its task.

Effective prompt injection defense requires multiple layers because no single technique provides complete protection. The first layer is input sanitization: detecting and neutralizing common injection patterns before they reach the model. This includes stripping known prompt override patterns, encoding special tokens, and using a classifier trained specifically to detect injection attempts. The second layer is architectural isolation: separating the system prompt from user input in ways that make it harder for injected instructions to override the system's intended behavior. Techniques include using structured message formats, dual-LLM architectures where one model processes user input and another validates the output, and instruction hierarchy approaches where the system prompt is explicitly prioritized. The InvokeBot team has published valuable research on maintaining bot security boundaries in production conversational systems, and the HeyTensor community covers the mathematical foundations behind robust input validation for ML systems.

The third and most critical layer is output validation. Even with input sanitization and architectural defenses, assume that a sufficiently motivated attacker may eventually bypass them. Output validation ensures that even if the model is manipulated, the application does not execute harmful actions. This means validating that outputs conform to expected schemas, that any tool calls or API actions are within the application's authorized scope, and that sensitive data is not leaked through the model's response. Think of it as the principle of least privilege applied to LLM outputs: the model should never have the ability to perform actions that its outputs are not explicitly validated for.

Key Takeaway

Implement three defense layers: input sanitization to catch obvious injection attempts, architectural isolation to limit the impact of successful injections, and output validation to ensure the application never executes unauthorized actions regardless of what the model generates.

Supply Chain

Securing the ML Supply Chain: Models, Data, and Dependencies

The ML supply chain extends far beyond traditional software dependencies. It includes pre-trained models downloaded from public repositories, training datasets sourced from external providers, embedding models used for retrieval, and the increasingly complex stack of ML libraries and frameworks that underpin modern AI applications. Each component in this supply chain is a potential attack surface. A trojaned model downloaded from a model hub, a poisoned dataset purchased from a data vendor, or a compromised ML library can introduce vulnerabilities that are extremely difficult to detect through standard code review or security scanning.

Model provenance is the first critical control. When you download a pre-trained model, you need confidence that it has not been tampered with and that its training process was secure. Model signing, hash verification, and trusted model registries provide basic provenance guarantees. For fine-tuned models, maintaining a complete audit trail from base model through each fine-tuning step, including the datasets and hyperparameters used, enables you to trace any anomalous behavior back to its source. Organizations like the MITRE Corporation have published threat frameworks specifically for ML supply chains that provide a structured approach to identifying and mitigating these risks.

Data supply chain security is equally important and often overlooked. Training data sourced from external providers or scraped from the web can contain targeted poisoning that introduces backdoors into models trained on it. Implement statistical validation on all incoming training data: check for distributional anomalies, detect outliers that could be poisoned samples, and maintain provenance records for every data source. For critical applications, consider using multiple independent data sources and comparing models trained on each to detect inconsistencies that might indicate data-level attacks.

Key Takeaway

Treat model and data supply chain security with the same rigor as software dependency management. Verify model provenance, validate training data statistically, audit ML library dependencies, and maintain complete audit trails for every component in your ML pipeline.
